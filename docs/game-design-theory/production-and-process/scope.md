# 7.5.1 Scope

## Classification

**Type:** Design / Engineering Guide

**Category:** Making Games

# Design Problem

Imagine deciding to make a game.

You begin writing ideas.

Soon your notebook contains:

- fifty enemies
- one hundred weapons
- crafting
- multiplayer
- procedural generation
- voice acting
- hundreds of quests
- twenty bosses

The project sounds exciting.

It also becomes impossible to finish.

One of the most important skills in game development is not adding ideas.

It is deciding **which ideas belong in the game right now.**

This process is called **Scope**.

---

# Why This Matters

Every project has limits.

Examples include:

- time
- experience
- budget
- artwork
- programming
- testing

Good designers work within those limits.

Rather than attempting to build everything...

...they focus on building something complete.

Finished games teach far more than unfinished ambitions.

---

# Mental Model

Imagine building a house.

You could begin by designing:

- a castle
- a skyscraper
- an entire city

Or...

you could begin with one room.

Complete it.

Then build another.

Games work the same way.

```text
Small Complete System

↓

Finished Game

↓

Larger Projects
```

Completing projects builds experience.

Experience expands future possibilities.

---

# Examples

Imagine two student projects.

### Project A

- 40 enemy types
- 100 items
- online multiplayer
- procedural worlds
- incomplete after six months

---

### Project B

- one player
- three enemies
- one complete level
- polished gameplay
- finished

Which project teaches more about game development?

Often...

the finished project.

---

# Dissecting the Idea

Notice the relationship.

```text
Ideas

↓

Choose

↓

Build

↓

Finish
```

Scope is not about limiting creativity.

It is about choosing where creativity should be invested.

Every project leaves ideas for the next project.

---

# Design Principle

## Finish something.

Small completed games usually teach more than enormous unfinished ones.

A finished project provides experience with:

- programming
- design
- testing
- iteration
- polish

Finishing is itself a design skill.

---

# Practical Observation

Beginning designers often estimate projects by counting ideas.

Experienced designers often estimate projects by counting work.

For example:

```text
One Enemy

↓

Animation

↓

Sound

↓

AI

↓

Testing

↓

Balancing
```

Even simple features often require much more work than they first appear.

Understanding scope improves planning.

---

# Common Misconceptions

### "A bigger game is a better game."

Not necessarily.

Many memorable games are built around a remarkably small number of carefully designed ideas.

---

### "Removing features means the game becomes worse."

Sometimes removing features makes the remaining systems much stronger.

Focus often improves quality.

---

### "Planning limits creativity."

Good planning creates room for creativity by reducing unnecessary complexity.

---

### "I'll add polish later."

Polish requires time.

Projects that exceed their scope often reach the deadline before polishing ever begins.

---

# Reflection

Imagine you have six weeks to build a game.

Which of the following would you reduce first?

- number of enemies?
- number of levels?
- number of mechanics?
- artwork?
- story?

Which parts of the game are absolutely essential?

Which ideas could become future projects instead?
