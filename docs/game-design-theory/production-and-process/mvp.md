# 7.5.2 Minimum Viable Product (MVP)

## Classification

**Type:** Design / Engineering Guide

**Category:** Making Games

**Difficulty:** Beginner

**Estimated Time:** 20–30 minutes

---

# Design Problem

Suppose you want to build a platformer.

Eventually your game might contain:

- twenty levels
- dozens of enemies
- power-ups
- collectibles
- music
- cutscenes
- boss fights

Where should you begin?

Many beginning developers start by building everything at once.

Professional developers often begin somewhere much smaller.

They ask:

> **What is the smallest version of this game that proves the idea works?**

This is called the **Minimum Viable Product**, or **MVP**.

---

# Why This Matters

An MVP is not a demo.

It is not a prototype filled with placeholder systems.

It is the smallest **complete** version of the game that captures its central experience.

Building an MVP allows designers to answer an important question early:

> **Is this game actually fun?**

If the answer is no...

it is much easier to change a small project than a large one.

---

# Mental Model

Think about learning to bake.

You would not begin by preparing a five-course meal.

You might begin with one loaf of bread.

If the bread works...

you can build upon it.

Games work the same way.

```text
Core Idea

↓

Small Complete Game

↓

Expand
```

Every new feature grows from a working foundation.

---

# Example

Imagine designing a platformer.

Your final vision might include:

- ten worlds
- fifty enemies
- multiple abilities
- collectibles
- bosses

Your MVP might contain only:

- one level
- one enemy
- one collectible
- one goal

If that one level is enjoyable...

the rest of the project becomes much easier to build with confidence.

---

# Dissecting the Idea

Notice the order.

Many beginning projects attempt:

```text
Features

↓

Features

↓

Features

↓

Hope It Is Fun
```

An MVP reverses the process.

```text
Core Experience

↓

Test

↓

Improve

↓

Expand
```

The game earns its complexity.

---

# Design Principle

## Prove the idea before expanding it.

Every additional feature increases development time.

Before adding more content...

first demonstrate that the central gameplay experience is enjoyable.

A strong foundation supports everything that follows.

---

# Practical Observation

When planning an MVP, ask yourself:

> If I removed everything except the heart of the game...

...what would remain?

For a platformer:

- movement
- jumping
- reaching the goal

For a puzzle game:

- one puzzle mechanic

For an RPG:

- one meaningful encounter

If those experiences are enjoyable...

the project has a strong foundation.

---

# Common Misconceptions

### "MVP means unfinished."

No.

An MVP is intentionally small.

It should still feel complete.

---

### "The MVP should contain every planned mechanic."

Usually the opposite.

The MVP contains only the mechanics required to demonstrate the core experience.

Everything else comes later.

---

### "Polish is unnecessary."

Even a small MVP benefits from:

- responsive controls
- clear feedback
- thoughtful pacing

Players judge the experience, not the number of features.

---

### "The MVP becomes the final game."

Not necessarily.

The MVP is the beginning.

Successful ideas naturally grow from it through testing and iteration.

---

# Reflection

Imagine your next game project.

What is the smallest version that would still feel like the game you want to build?

Which features are:

- essential?
- useful?
- optional?

Could you remove half of your planned features and still communicate the core idea?

# Looking Back

Earlier we explored **Scope**.

Scope asks:

> **What can I realistically build?**

MVP asks a different question:

> **What must I build first?**

Together they help transform ambitious ideas into achievable projects.

Small, complete experiences often become the strongest foundations for larger games.
