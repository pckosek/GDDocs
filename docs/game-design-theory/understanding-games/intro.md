# 7.1.1 What is a Game?

## Classification

**Type:** Design / Engineering Guide

**Category:** Understanding Games

**Difficulty:** Beginner

**Estimated Time:** 20–30 minutes

# Design Problem

Suppose someone asks:

> "What kind of game are you making?"

At first, this seems like an easy question.

But before we can answer *what kind* of game it is...

...we must first answer something more fundamental.

> **What is a game?**

Consider the following.

- Chess
- Minecraft
- Tetris
- Hide and Seek
- Football
- Sudoku
- SimCity

These experiences feel very different.

Yet we instinctively recognize them all as games.

What do they have in common?

---

# Why This Matters

Understanding what makes something a game helps us design better ones.

When creating a game we are making decisions about:

- goals
- rules
- player actions
- challenges
- feedback
- rewards

These ideas appear in almost every genre.

Learning this vocabulary allows us to describe, compare, and improve our own designs.

---

# A Working Definition

Although there are many definitions of games, most share several common characteristics.

A game typically contains:

- one or more players
- rules
- goals
- meaningful choices
- feedback
- challenge

These elements work together to create an interactive experience.

Not every game emphasizes them equally.

But most games contain some combination of them.

---

# Mental Model

Think about a board game.

```text
Rules

↓

Player Choices

↓

Consequences

↓

Progress

↓

Goal
```

Now imagine a digital game.

The same pattern still exists.

The technology changes.

The underlying structure remains remarkably similar.

---

# Dissecting the Definition

Notice that a game is more than software.

A game is not simply:

```text
Graphics

+

Sound

+

Code
```

Instead...

those systems work together to create:

```text
Challenge

↓

Decision

↓

Feedback

↓

Experience
```

Programming builds the systems.

Game design shapes the experience.

---

# Design Principle

## Games create meaningful interaction.

Players do more than observe.

They make decisions.

Those decisions produce consequences.

Those consequences create the experience of play.

Interaction is one of the defining characteristics of games.

---

# Practical Observation

Many beginning designers focus on features.

For example:

- swords
- inventory
- crafting
- multiplayer

Experienced designers often begin somewhere different.

They ask:

> **What will the player actually do?**

Understanding the player's experience usually leads to stronger designs than simply adding more features.

---

# Common Misconceptions

### "Games must have winners."

Not necessarily.

Many games focus on exploration, creativity, or storytelling rather than winning.

---

### "Games must contain enemies."

No.

Many games contain no combat at all.

Players may instead solve puzzles, build cities, tell stories, or simply explore.

---

### "Games are defined by their graphics."

Graphics influence the player's experience.

They do not define the game itself.

The same game could be represented using text, simple shapes, or detailed artwork.

---

### "Programming and game design are the same."

Programming builds the systems.

Game design determines how those systems create meaningful experiences for players.

Both are essential.
