# Educational Workbench Documentation Constitution

*A Living Document*

## Preamble

This repository is an extension of the classroom, not a replacement for it.

Its purpose is not merely to document Godot, Python, or game development techniques. Its purpose is to preserve momentum between class meetings, reduce unnecessary frustration, and empower students to continue exploring independently.

Every page should leave a student more capable of creating something than when they arrived.

This documentation is written for students who are curious, students who are struggling, students who missed class, students who want to read ahead, and students who find themselves debugging a project late at night. It exists so that progress does not stop simply because the instructor is no longer in the room.

The repository is a workbench.

Students should return to it repeatedly throughout the course, and ideally long after the course has ended.

---

# I. Educational Philosophy

Learning is an act of exploration.

Students are not expected to memorize every operation, every API call, or every editor feature. Instead, they should become increasingly confident explorers who understand how to investigate problems, test ideas, and build meaningful projects.

The purpose of instruction is not to eliminate challenge.

The purpose is to eliminate unnecessary barriers so that students can spend their effort on solving interesting problems rather than fighting the tools.

Projects provide the context that gives meaning to exploration.

Every concept taught in this repository should ultimately contribute toward creating something tangible.

Students should leave this course not simply knowing how to write code, but thinking like creative practitioners—people who continue refining their work after it technically functions because they recognize that compiling is the beginning of craftsmanship, not the end.

---

# II. Purpose of the Repository

This repository exists to:

* support classroom instruction without replacing it;
* preserve continuity between class meetings;
* provide reliable reference material for fundamental operations and techniques;
* reduce unnecessary cognitive load while learning unfamiliar tools;
* encourage experimentation through complete, working examples;
* become a long-term reference students can revisit months or years later.

This repository is intentionally designed to answer practical questions quickly.

A student working late at night should be able to locate a relevant page, find a working example, understand why it works, and continue making progress without unnecessary searching.

---

# III. Audience

The primary audience is high school students enrolled in an advanced game development course.

Students are expected to possess prior programming experience but are not expected to have prior experience with Godot.

Students arrive with widely varying backgrounds.

Some have extensive programming experience.

Others have only completed prerequisite coursework.

Some are already creating games independently.

Others are encountering a game engine for the first time.

The documentation must serve all of them without becoming overwhelming.

It should remain approachable while still providing depth for students who choose to explore further.

---

# IV. Documentation Philosophy

Documentation exists to support exploration.

Every page should reduce uncertainty without removing opportunities for discovery.

Students should never become stuck because of missing mechanical details.

Whenever possible, examples should include the complete sequence of operations necessary to reproduce the demonstrated result.

If students are expected to click buttons, create nodes, configure inspector properties, or arrange scene hierarchies, those actions should be documented clearly.

The documentation should minimize ambiguity while maximizing student agency.

---

# V. Examples

Examples are among the most valuable resources in this repository.

Every example should be:

* technically correct;
* immediately usable;
* copy-paste ready when appropriate;
* minimal while remaining complete;
* directly relevant to the surrounding discussion.

Examples exist to reduce unnecessary uncertainty.

Students should not spend valuable time wondering whether they omitted a required node, forgot a property, or misunderstood the editor.

Where visual context improves understanding, screenshots, scene trees, diagrams, and annotated interface elements should accompany the explanation.

---

# VI. Organization

The repository is organized around ideas rather than chapters.

Different pages serve different purposes.

Examples include:

* concepts;
* techniques;
* editor operations;
* Python language features;
* Godot features;
* debugging strategies;
* common mistakes;
* reference material.

Not every page follows an identical template.

Instead, each page should adopt the structure that best serves its educational purpose while maintaining consistent navigation and presentation.

Every page should clearly communicate:

* what problem it solves;
* when students should use it;
* how it relates to other topics;
* where students might continue next.

---

# VII. Writing Style

The writing should sound like an experienced teacher working beside a student.

It should be conversational without becoming casual.

Friendly without becoming performative.

Professional without becoming academic.

Explanations should be concise but complete.

Paragraphs should develop coherent ideas rather than fragmenting thoughts into isolated single-sentence blocks.

Whenever appropriate, the author's experience may appear naturally:

> Students often run into this issue...

> One thing I've noticed...

> This tends to be confusing at first...

These observations should feel authentic rather than scripted.

---

# VIII. Authenticity

Authenticity is essential.

Examples should represent techniques students are expected to understand.

Code should never be included simply because it works.

Students should leave examples with greater understanding than when they began.

Documentation should acknowledge simplifications honestly.

If a technique is intentionally simplified for instructional purposes, the documentation should say so explicitly.

Students should be encouraged to build ownership of their work rather than relying unquestioningly on copied solutions.

Artificial intelligence is a tool for exploration, not a substitute for thinking.

Its value lies in supporting investigation, reinforcing understanding, and extending curiosity—not replacing the learning process itself.

---

# IX. Design Principles

Good design supports learning.

Visual organization should reduce cognitive load.

Navigation should make information easy to locate.

Titles should be descriptive and searchable.

Callouts, collapsible sections, screenshots, diagrams, and illustrations should be used intentionally to emphasize important ideas without overwhelming the page.

The repository should feel inviting rather than dense.

Every design decision should help students find answers more quickly.

---

# X. Definition of Done

A page is complete when:

* its educational purpose is clear;
* its examples function correctly;
* necessary steps are fully documented;
* unnecessary distractions have been removed;
* related topics are referenced appropriately;
* students can confidently continue working after reading it.

Completeness is measured by student usefulness rather than word count.

---

# XI. Guiding Principle

Every revision should ask one question:

> **Does this make it easier for a student to continue exploring?**

If the answer is yes, the change supports the mission of the repository.

If the answer is no, the change should be reconsidered.
